## Exit To Start Menu Checklist

Note (from checklist research, not a task): the string `"Quit to Desktop"` also appears in
`docs/design/settings-menu.md` and `docs/checklists/settings-menu.md`. Both describe an
already-completed past feature as it existed at the time it shipped — they are historical
records, not live documentation. They are intentionally **not** included below; do not edit
them to say "Exit to Title" (`docs/design/exit-to-start-menu.md` itself, the current design
doc, also contains the old string in its motivation section — also intentionally not edited,
per "do not modify the design doc"). `README.md` has no mentions of "Quit to Desktop" or the
settings menu at all, so no README changes are needed.

- [x] Task A — `core/lua/sound.lua` — Add a new function `Sound.fade_music_group(group, target_vol, duration)`, placed after the existing `Sound.fade_music` function and before `Sound.stop_music`. Body: iterate over the module's internal `_music_tracks` table; for every entry whose `entry.group == group` **and** `entry.playing_intent == true`, set exactly these three fields (same three fields `Sound.fade_music` sets on a single entry): `entry.fade_target = target_vol`, `entry.fade_rate = (target_vol - entry.fade_vol) / duration`, `entry.stop_on_done = (target_vol == 0)`. Do **not** call the existing `_claim_group` helper, and do **not** add the "start playing if not already playing" branch that `Sound.fade_music` has for `target_vol > 0` — this new helper only ever fades OUT tracks that are already mid-playback in a group (the caller doesn't know which specific track in the group is active), so entries with `playing_intent == false` must be left completely untouched. This reuses the same `Sound.update` fade machinery that already drives `fade_vol`/`fade_target`/`stop_on_done`/`stop()`-on-completion for every other track. Exact call signature another task will use: `Sound.fade_music_group("bg", 0, 2)`. Also add test coverage to `tests/test_sound.lua` (append near the end, following the file's existing `do ... end` block + `print("PASS: ...")` convention, and its pattern of stubbing `love.filesystem.getInfo` to control which tracks load): (1) load the module's existing `MANIFEST` (bg1/bg2 both group `"bg"`), call `Sound.play_music("bg1")` (playing_intent becomes true), then call `Sound.fade_music_group("bg", 0, 1)` followed by `Sound.update(1.5)` (past the 1s duration), and assert `Sound.is_music_playing("bg1") == false` afterward — proving the group-wide fade-out actually reaches 0 and stops the source (mirrors the `stop_on_done` behavior already exercised elsewhere in this file for `Sound.fade_music`). (2) Prove tracks in the group that were never started are left alone: after (1), also load with `bg2` present but never call `Sound.play_music("bg2")`/`Sound.fade_music("bg2", ...)`, call `Sound.fade_music_group("bg", 0, 1)` again, `Sound.update(1.5)`, and assert no error occurs (bg2's `playing_intent` was already false, so it must not be touched — the existing per-track stub pattern used elsewhere in this file, e.g. stubbing `src.stop` to count calls, can be reused to assert bg2's `stop()` is never called if you want a stronger assertion). (3) Call `Sound.fade_music_group("nonexistent_group", 0, 1)` then `Sound.update(0.016)` and assert it runs without error (silent no-op for an unknown group name, matching how `Sound.fade_music("nonexistent", ...)` already behaves elsewhere in this file).

- [x] Task B — `game/scenes/start_scene.lua` — Two changes, both needed together so the exit-to-title path (built in Task C, a separate file) doesn't get its music fade stomped when it switches to a freshly-constructed `StartScene`. (1) In `StartScene.new()`, add a new field `self.is_title_scene = true` immediately next to the existing `self.esc_opens_settings = true` line — do not remove, rename, or repurpose `esc_opens_settings`; both fields coexist. `main.lua` (Task C) will read `manager.current.is_title_scene` to detect "the active scene is the title screen" — this is the exact field name it expects. (2) In `StartScene:on_enter()`, change the unconditional `Sound.play_music("menu")` call to be guarded: `if not Sound.is_music_playing("menu") then Sound.play_music("menu") end`. Reason (this is a real interaction bug found during design research, not present in the approved design doc's text, so it needs to be handled here explicitly): the exit-to-title path in `main.lua` (Task C) calls `Sound.fade_music("menu", 1, 2)` — which internally starts the source playing at volume 0 and ramps it up — *before* calling `manager:switch(StartScene.new(...))`. By the time the new `StartScene:on_enter()` runs, `Sound.is_music_playing("menu")` will already be `true` (the fade already started it playing), so the guard must skip the old unconditional `Sound.play_music("menu")` call — calling it anyway would hard-set the track to full volume instantly, canceling the in-progress fade-in animation (and could produce an audible pop/restart since `Sound.play_music` also calls `src:play()` again). At normal game boot, the menu track is not yet playing (`Sound.is_music_playing("menu")` is `false`), so the guard is a no-op there and `Sound.play_music("menu")` still runs exactly as it does today — boot behavior is unchanged. Also add test coverage to `tests/test_start_scene.lua` (append after the existing `-- ── on_enter plays menu music ──` block, following that block's pattern of stubbing `Sound.play_music`): (1) assert `scene.is_title_scene == true` right where the existing `assert(scene.esc_opens_settings == true, ...)` line is, in the "construction" test block. (2) A new test block: stub both `Sound.play_music` (to record calls, as the existing test does) and `Sound.is_music_playing` to return `true` for `"menu"`, construct a `StartScene`, call `scene:on_enter()`, and assert `Sound.play_music` was **not** called — proving the guard skips the hard-play when menu is already playing. Restore both stubbed functions afterward, matching the existing test's `orig_play_music` restore pattern.

- [x] Task C — `game/scenes/settings_menu.lua` **and** `main.lua` — Handled as one task because both files share a new callback contract (name and 0-argument signature) that must match exactly; splitting this across two isolated agents risks a mismatch. **In `game/scenes/settings_menu.lua`:** (1) In the top-level `ITEMS` table (currently `local ITEMS = { "Fullscreen / Window", "Keybinds", "Exit Settings", "Quit to Desktop" }`), change the 4th string from `"Quit to Desktop"` to `"Exit to Title"`. The label stays this same text regardless of where the menu was opened from (no context-dependent text). (2) Change the constructor signature from `function SettingsMenu.new(settings_state, input, on_close)` to `function SettingsMenu.new(settings_state, input, on_close, on_exit_to_title)`, and store the new parameter as `self._on_exit_to_title = on_exit_to_title`, added right next to the existing `self._on_close = on_close` line. (3) In `SettingsMenu:_confirm()`, replace the current item-4 branch — `elseif self.selected == 4 then if self._on_close then self._on_close() end love.event.quit() end` — with:
  ```lua
  elseif self.selected == 4 then
      self.is_open = false
      if self._on_exit_to_title then
          self._on_exit_to_title()
      end
  end
  ```
  Do not call `self._on_close` here, and do not call `love.event.quit()` directly — the new `on_exit_to_title` callback (implemented in `main.lua`, below) now owns both responsibilities (saving settings, and deciding whether to quit outright or switch back to the title screen). Setting `self.is_open = false` here closes the pause overlay immediately, so it stops rendering on top of whatever scene ends up active after the callback runs. **In `main.lua`:** (1) In `love.load()`, add a new local function right after the existing `local function on_close() Save.write_settings(ss:to_save()) end` block:
  ```lua
  local function on_exit_to_title()
      Save.write_settings(ss:to_save())
      if manager.current and manager.current.is_title_scene then
          love.event.quit()
      else
          Sound.fade_music_group("bg", 0, 2)
          Sound.fade_music("menu", 1, 2)
          manager:switch(StartScene.new(manager, ss, input))
      end
  end
  ```
  `Sound.fade_music_group(group, target_vol, duration)` is a new function added to `core/lua/sound.lua` in a separate task (Task A) — call it exactly as shown above (group `"bg"`, target `0`, duration `2`). `manager.current.is_title_scene` is a new field added to `StartScene` in a separate task (Task B) — it is `true` only on `StartScene` instances; read it exactly as shown, with no other type check. The duration `2` matches the existing forward crossfade's duration used in `GameScene:on_enter()` (`Sound.fade_music("menu", 0, 2)` / `Sound.fade_music(bg_track, 1, 2)`). `StartScene`, `Save`, and `Sound` are already `require`d at the top of `main.lua` — no new requires are needed. Call the fade calls **before** `manager:switch(...)`, in the exact order shown (bg fade-out, then menu fade-in, then switch) — this ordering matters together with Task B's `on_enter` guard: `Sound.fade_music("menu", 1, 2)` must run, and thus start the menu track playing, before `StartScene:on_enter()` checks `Sound.is_music_playing("menu")`. (2) Change the `settings_menu = SettingsMenu.new(ss, input, on_close)` call to `settings_menu = SettingsMenu.new(ss, input, on_close, on_exit_to_title)`. **Tests:** add a new file `tests/test_settings_menu.lua` (none currently exists; follow the stubbing style used in `tests/test_start_scene.lua`, and note `game/settings_state.lua`'s `SettingsState.new()` is a real, directly-usable constructor — no fake/mock object needed, as `tests/test_settings_state.lua` demonstrates). Required cases: (1) Construct `SettingsMenu.new(SettingsState.new(), nil, function() end, function() exit_calls = exit_calls + 1 end)`, stub `love.event.quit` to count calls, set `menu.is_open = true` and `menu.selected = 4`, call `menu:_confirm()`, then assert `exit_calls == 1`, assert the stubbed `love.event.quit` call count is `0` (item 4 must no longer quit directly), and assert `menu.is_open == false`. Restore the real `love.event.quit` afterward. (2) Construct a `SettingsMenu` with `on_exit_to_title` passed as `nil` (4th arg omitted), set `selected = 4`, call `menu:_confirm()`, and assert it does not error (defensive nil-callback guard). Note: `ITEMS` is a private local table not exposed by the module, so the "Exit to Title" label text itself is not practical to assert from an external test file — skip asserting the label text; that part is verified visually/by code review instead. Note: `main.lua`'s `on_exit_to_title` quit-vs-switch branching itself is glue code in the LÖVE entry point (`main.lua` is not `require`-able as a plain module without triggering `love.load()`'s side effects) — this codebase has no `test_main.lua` and no precedent for unit-testing `main.lua` directly, so do not attempt to add one; that logic is covered by manual/visual verification only.
